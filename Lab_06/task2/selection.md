# Выбор паттернов для сценариев

1. **SCENARIO_01 (App Config Store):**
   * **Семья:** Creational (Порождающие)[cite: 1413].
   * [cite_start]**Паттерн:** Singleton[cite: 1305].
   * [cite_start]**Обоснование:** Требуется ровно один экземпляр хранилища настроек на все приложение[cite: 1005].
   * **Альтернатива:** Module Pattern. Он тоже дает один экземпляр, но Singleton лучше контролирует момент создания.

2. **SCENARIO_02 (Payment Gateway Adapter):**
   * **Семья:** Structural (Структурные)[cite: 1415].
   * [cite_start]**Паттерн:** Adapter[cite: 1307].
   * [cite_start]**Обоснование:** Нужно состыковать интерфейс современного кода с несовместимым старым банковским API [cite: 1041-1043, 1312].
   * **Альтернатива:** Facade. Но Фасад упрощает сложную систему, а Адаптер меняет интерфейс одного конкретного класса.

3. **SCENARIO_03 (UI Notifications):**
   * **Семья:** Behavioral (Поведенческие)[cite: 1416].
   * [cite_start]**Паттерн:** Observer[cite: 1309].
   * [cite_start]**Обоснование:** Система должна уведомлять множество независимых компонентов (GradeBook, Calendar) о событии сдачи задания [cite: 1069-1072].
   * **Альтернатива:** Mediator. Но здесь важнее подписка на изменения, чем централизованное управление логикой.

4. **SCENARIO_04 (Report Builder):**
   * **Семья:** Creational (Порождающие)[cite: 1413].
   * [cite_start]**Паттерн:** Builder[cite: 1305].
   * [cite_start]**Обоснование:** Процесс сборки отчета одинаковый, но представления (HTML/PDF) разные [cite: 1105-1116].
   * **Альтернатива:** Factory Method. Но Factory создает объект за один шаг, а Builder — пошагово.

5. **SCENARIO_05 (Undo/Redo):**
   * **Семья:** Behavioral (Поведенческие)[cite: 1416].
   * [cite_start]**Паттерн:** Command[cite: 1355].
   * [cite_start]**Обоснование:** Каждое действие пользователя должно быть объектом с методами execute и undo для поддержки истории [cite: 1150-1157].
   * **Альтернатива:** Memento. Но Memento сохраняет состояние, а Command сохраняет именно действие.

6. **SCENARIO_06 (Particle System):**
   * **Семья:** Structural (Структурные)[cite: 1415].
   * [cite_start]**Паттерн:** Flyweight[cite: 1307].
   * [cite_start]**Обоснование:** Нужно отрисовать 50,000 частиц, разделяя общие данные (цвет, форма) для экономии памяти [cite: 1218-1232].
   * **Альтернатива:** Prototype. Но клонирование 50,000 объектов не решит проблему потребления памяти так эффективно.
